# Правила крита и бонусов

Документ описывает будущую систему бонусов для RPM-боя. Это дизайн-правила, а не утверждение, что все поля уже реализованы в коде.

## Базовый крит

Крит должен остаться не случайным, а зависящим от действия игрока или бота.

Крит происходит, если атакующий:

- летит в сторону цели достаточно быстро;
- попадает своей передней зоной;
- попадает в уязвимую зону цели: обычно бок или спину.

Текущее простое правило:

```ts
attackerZone === "front"
targetZone !== "front"
hitSpeed >= criticalSpeed
```

Улучшенное правило должно использовать параметры конкретного атакующего:

```ts
criticalSpeed =
  attackerEffectiveMaxMoveSpeed
  * baseCriticalSpeedRatio
  * (1 - attacker.critSpeedEase)
```

Рекомендуемые значения:

```ts
baseCriticalSpeedRatio = 0.95
critSpeedEase = 0
```

Смысл:

- без бонусов крит требует примерно 95% от текущей доступной максимальной скорости атакующего;
- если RPM упал и доступная максимальная скорость стала ниже, порог крита тоже становится ниже;
- у игрока и бота разные пороги, потому что они считаются от их собственной скорости.

Примеры:

```ts
player: 17 * 0.95 = 16.15
bot:    14 * 0.95 = 13.30
```

## Бонус шанса крита

В этой системе "+20% крита" не должен быть случайным шансом. Лучше считать это как "крит легче выполнить".

Бонус должен храниться на конкретном атакующем:

```ts
attacker.critSpeedEase
attacker.critZoneEase
```

Так один спиннер может чаще критовать, но не меняет правила для остальных.

### Упрощение скорости крита

`critSpeedEase` снижает только порог скорости для этого атакующего.

```ts
criticalSpeed *= 1 - attacker.critSpeedEase
```

Пример:

```ts
critSpeedEase = 0.2
normal threshold = 16.15
bonus threshold = 16.15 * 0.8 = 12.92
```

То есть "+20% крита" значит: этому спиннеру нужно на 20% меньше скорости, чтобы выполнить крит.

### Упрощение зон крита

`critZoneEase` делает проверку зон мягче только для крита этого атакующего. Обычные зоны урона для всех не меняются.

Сейчас зоны определяются через `dot product`:

```ts
front if dot > 0.5
back  if dot < -0.5
side  otherwise
```

Для проверки крита можно использовать отдельные пороги:

```ts
attackerFrontThreshold = 0.5 * (1 - attacker.critZoneEase)
targetProtectedFrontThreshold = 0.5 + 0.5 * attacker.critZoneEase
```

Потом:

```ts
attackerCanCritFromZone = attackerDot > attackerFrontThreshold
targetIsCritVulnerable = targetDot <= targetProtectedFrontThreshold
```

Пример с `critZoneEase = 0.2`:

```ts
attackerFrontThreshold = 0.4
targetProtectedFrontThreshold = 0.6
```

Результат:

- передняя зона атакующего чуть шире только для проверки крита;
- защищенная передняя зона цели чуть уже только против этого атакующего;
- остальные спиннеры продолжают использовать обычные пороги.

Для первой версии лучше начать только с `critSpeedEase`. Если бонус крита будет ощущаться слабо, потом добавить `critZoneEase`.

## Бонус критического урона

Бонус критического урона не должен повышать частоту крита. Он применяется только после того, как крит уже подтвержден.

Базовое правило:

```ts
criticalDamageMultiplier = 1.5
```

Рекомендуемое правило бонуса:

```ts
finalCriticalDamageMultiplier =
  criticalDamageMultiplier
  * attacker.critDamageMultiplier
```

Значение по умолчанию:

```ts
critDamageMultiplier = 1
```

Пример для "+20% критического урона":

```ts
finalCriticalDamageMultiplier = 1.5 * 1.2 = 1.8
```

То есть удар не критует чаще, но подтвержденный крит наносит больше урона.

## Бонус скорости

Бонус скорости увеличивает доступную максимальную скорость конкретного спиннера.

```ts
attackerEffectiveMaxMoveSpeed =
  attacker.baseMoveSpeed
  * attacker.speedMultiplier
  * attacker.rpmMoveSpeedRatio
```

Значение по умолчанию:

```ts
speedMultiplier = 1
```

Важное правило: порог скорости для крита должен использовать тот же `attackerEffectiveMaxMoveSpeed`.

Значит бонус скорости не делает крит автоматически легче. Если игрок получает +20% скорости, его порог скорости для крита тоже растет на 20%.

Пример:

```ts
base speed = 17
speedMultiplier = 1.2
effective max speed = 20.4
criticalSpeed = 20.4 * 0.95 = 19.38
```

Игрок стал быстрее, но для крита все еще должен лететь примерно на 95% от своей доступной скорости.

Если нужно, чтобы бонус скорости еще и помогал критовать, это должен делать отдельный бонус `critSpeedEase`, а не сам `speedMultiplier`.

## Бонус лечения max RPM

Лечение max RPM восстанавливает `maxRPM`, который был потерян от урона.

Рекомендуемое правило:

```ts
spinner.maxRPM = Math.min(
  spinner.absoluteMaxRPM,
  spinner.maxRPM + maxRpmHealAmount,
)
```

Бонус не должен восстанавливать выше `absoluteMaxRPM`.

Рекомендуемое поведение для первой версии:

- восстанавливать только `maxRPM`;
- не восстанавливать `currentRPM` мгновенно;
- дать движению восстановить `currentRPM` до нового `maxRPM` обычным способом.

Так лечение не становится мгновенным ускорением. Если позже нужен моментальный хил текущего состояния, лучше добавить отдельный бонус для `currentRPM`.

## Проверка на противоречия

Системы не конфликтуют, если все бонусы хранятся на конкретном спиннере и применяются в таком порядке:

```ts
1. считаем attackerEffectiveMaxMoveSpeed
2. считаем hitSpeed от скорости атакующего в сторону цели
3. считаем criticalSpeed от текущей доступной скорости атакующего
4. применяем attacker.critSpeedEase к criticalSpeed
5. применяем attacker.critZoneEase только к проверке зон крита
6. если крит подтвержден, применяем attacker.critDamageMultiplier
7. maxRPM-heal применяем только к спиннеру, который получил лечение
```

Во время игры нельзя менять глобальные пороги зон или глобальный `criticalSpeed` для всех. Бонусы должны лежать на владельце бонуса.

Итог:

- один спиннер может критовать легче, не усиливая всех остальных;
- бонус скорости увеличивает скорость, но не ломает частоту критов;
- бонус критического урона усиливает только уже подтвержденный крит;
- лечение `maxRPM` восстанавливает долгосрочную живучесть, но не дает мгновенный скачок текущей скорости.
